# 260305 프로젝트 구조 통합 - 빌드/배포 문제 근본 해결 플랜

## 문제 요약

LID 프로젝트는 루트(`/`)와 프론트엔드(`frontend/`)가 분리되어 있어 Firebase 설정이 양쪽에 흩어져 있고, 빌드/배포 시마다 경로 꼬임 문제가 반복됨.

**SEL HUB**(정상 작동)과 **LID**(문제 발생)의 핵심 차이:

| 항목 | SEL HUB (정상) | LID (문제) |
|------|---------------|-----------|
| 프론트엔드 위치 | 루트 (`/src/`) | `frontend/src/` |
| firebase.json | 루트 1개 | 루트 + frontend/ 2개 |
| .firebaserc | 루트 1개 | frontend/ 에만 존재 |
| package.json | 루트 1개 | frontend/ 에만 존재 (루트에 없음) |
| vite.config | 루트 1개 | frontend/ 에만 존재 |
| 빌드 실행 위치 | 루트에서 `npm run build` | `cd frontend && npm run build` |
| 배포 실행 위치 | 루트에서 `firebase deploy` | 어디서 해야 하는지 애매 |
| firestore indexes | 1개 (일관됨) | 2개 (루트 vs frontend 내용 다름!) |

---

## 해결 목표

SEL HUB와 동일한 단일 루트 구조로 통합하여, 루트에서 `npm run build && firebase deploy`만 하면 끝나는 구조로 만든다.

---

## 작업 단계

### Phase 1: 프론트엔드 파일을 루트로 이동

#### 1-1. `frontend/src/` → 루트 `src/`로 이동
```
frontend/src/App.jsx        → src/App.jsx
frontend/src/main.jsx       → src/main.jsx
frontend/src/assets/        → src/assets/
frontend/src/components/    → src/components/
frontend/src/context/       → src/context/
frontend/src/firebase/      → src/firebase/
frontend/src/pages/         → src/pages/
frontend/src/services/      → src/services/
frontend/src/styles/        → src/styles/
frontend/src/utils/         → src/utils/
```

#### 1-2. `frontend/public/` → 루트 `public/`로 이동
```
frontend/public/            → public/
```

#### 1-3. `frontend/index.html` → 루트 `index.html`로 이동
```
frontend/index.html         → index.html
```

#### 1-4. `frontend/picture/` → 루트 `picture/`로 이동 (PWA 아이콘 등)
```
frontend/picture/           → picture/
```

### Phase 2: 설정 파일 통합

#### 2-1. `frontend/package.json` → 루트 `package.json` 생성
- `frontend/package.json` 내용을 루트로 이동
- scripts 수정:
  ```json
  {
    "scripts": {
      "dev": "vite",
      "build": "vite build",
      "preview": "vite preview",
      "deploy": "npm run build && firebase deploy --only hosting"
    }
  }
  ```

#### 2-2. `frontend/vite.config.js` → 루트 `vite.config.js`로 이동
- `picture/` 경로 등 상대 경로 확인 후 조정

#### 2-3. `frontend/eslint.config.js` → 루트 `eslint.config.js`로 이동

#### 2-4. Firebase 설정 통합
- **루트 `firebase.json`** 수정: `"public": "frontend/dist"` → `"public": "dist"`
- **루트 `.firebaserc`** 생성 (frontend/.firebaserc 내용 사용):
  ```json
  {
    "projects": {
      "default": "lid-publishing-helper"
    }
  }
  ```
- `frontend/firebase.json` 삭제 (루트 것으로 통합)
- `frontend/.firebaserc` 삭제 (루트로 이동)

#### 2-5. Firestore 인덱스 통합
- 루트 `firestore.indexes.json`과 `frontend/firestore.indexes.json`의 내용이 다름
- 두 파일의 인덱스를 **하나로 병합**하여 루트 `firestore.indexes.json`에 통합:
  ```json
  {
    "indexes": [
      { "comments": ["articleId ASC", "createdAt ASC"] },
      { "budgetCategories": ["year ASC", "createdAt ASC"] },
      { "expenses": ["year ASC", "date DESC"] }
    ]
  }
  ```

#### 2-6. `.gitignore` 통합
- 루트 `.gitignore`에 `frontend/.gitignore` 내용 병합
- `frontend/.gitignore` 삭제

### Phase 3: node_modules 재설치

#### 3-1. `frontend/node_modules/` 삭제
```bash
rm -rf frontend/node_modules frontend/package-lock.json
```

#### 3-2. 루트에서 의존성 설치
```bash
npm install
```
→ 루트에 `node_modules/`와 `package-lock.json` 생성

### Phase 4: 빌드/배포 검증

#### 4-1. 빌드 테스트
```bash
npm run build
```
→ 루트에 `dist/` 폴더 생성 확인

#### 4-2. 로컬 프리뷰
```bash
npm run preview
```
→ 브라우저에서 동작 확인

#### 4-3. Firebase 배포 테스트
```bash
firebase deploy --only hosting
```
→ https://lid-publishing-helper.web.app 정상 동작 확인

### Phase 5: 정리

#### 5-1. `frontend/` 폴더 삭제
- 모든 파일이 루트로 이동된 후 빈 폴더 제거
- **주의**: `frontend/dev-dist/`, `frontend/dist/` 등 빌드 산출물도 정리

#### 5-2. 루트 `.gitignore` 업데이트
```
dist/
dev-dist/
node_modules/
```

#### 5-3. `build-frontend.bat` 삭제
- 더 이상 frontend/ 서브디렉토리로 이동할 필요 없음

#### 5-4. 배포 가이드 업데이트
- `docs/배포_가이드.md`에서 `frontend/` 경로 참조 제거
- 새로운 단일 명령어 안내: `npm run build && firebase deploy`

## 통합 후 워크플로우

```
# 개발
npm run dev          ← 루트에서 바로 실행

# 빌드
npm run build        ← 루트에서 바로 실행, dist/ 생성

# 배포
npm run deploy       ← build + firebase deploy 한 방
```

**SEL HUB와 완전히 동일한 패턴.** `frontend/`로 cd할 필요 없고, firebase 설정 분산 없고, 경로 꼬임 없음.

---

## 리스크 및 주의사항

1. **import 경로**: `frontend/src/` 내부 import는 상대경로(`./`, `../`)를 쓰므로 폴더 구조가 그대로 유지되면 변경 불필요. 단, `vite.config.js`에 alias 설정이 있으면 확인 필요
2. **functions/**: `functions/` 폴더는 독립 Node.js 프로젝트이므로 변경 없음
3. **Git 히스토리**: 파일 이동이므로 git이 rename으로 감지. 대량 이동 시 한 커밋으로 처리 권장
4. **PWA 캐시**: 기존 사용자의 서비스 워커 캐시가 남아있을 수 있으나, 새 빌드 배포 시 자동 갱신됨
